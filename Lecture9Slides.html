Five good things about Jupyter notebooks
  1. Code and text together
  2. Plays nicely with graphics
  3. Excellent teaching tool
  4. Can use remotely
  5. Good for browsing/hacking
Four bad things about Jupyter notebooks
  1. Cells can run in any order
  2. Hard to tell new output from old
  3. Terrible for version control
  4. Hard to collaborate
Good things about (i)Python interpreter
  1. Code order fixed
  2. Widely available
  3. Hack, with history
Bad things about (i)Python interpreter
  1. Graphics can be tricky
  2. Fewer instructions
  3. mostly lines, not cells

PEP8 - Python style.

This code will run

import numpy as np; x=np.empty((10,10),float)
for i in range(10):
  for j in range(10):
    x[i,j]=i+2*j
for i in range(10):
    x[i,i]=i

But it's suprsisingly hard to read.

Python coding standard, PEP8 makes recommendations on well styled code, eg:

  • For spaces per block when indenting.
    • A space either side of = when assigning.
  • Add a space after a ,
  • Use blank lines (sparingly) to divide sections.

Other people do this so you should too. Your editor can help.

import numpy as np; x=np.empty((10,10),float)
for i in range(10):

		x[i, j] = i+2*j

	x[i,i]=i

Module Files :- Code reuse

There are lots of coding mnemonics:
  • KISS: Keep It Simple & Stupid
  • YAGNI: You ain’t gonna need it
  • DRY: Don't repeat yourself

In C, unit of shareable code is the library (.so/.dll file)

In Python, unit of sharable code is module file (.py). Also a text file, and might be useable as ascript

Counts as a module if you can import it.

Docstrings: Source documentation

When writing for reuse documentation is important

def factorial(x):
	"""Return the factorial function via recursion.

	Parameters
	----------
	x : int
		input integer

	Returns
	-------
	int
		factorial of x
	"""
	if x==1:
		return x
	else:
		return x*factorial(x-1)
